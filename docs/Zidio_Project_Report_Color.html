1. Project Overview

Mission: Build an end-to-end data science platform that ingests sales, customer, and inventory data to deliver accurate demand forecasts, customer segmentation, churn prediction, and inventory optimization recommendations for retail businesses.

Business Value Delivered

  • Reduced Stockouts: 30–50% reduction through accurate Prophet + LSTM ensemble forecasting.
  • Increased Revenue: 15–25% increase through better inventory decisions and targeted marketing.
  • Improved Retention: Proactive identification of at-risk customers via XGBoost classification.

Non-Functional Goals

  • Model Accuracy: MAPE ≤ 12% for demand forecasting
  • Latency: < 5 minutes for daily batch jobs via Apache Airflow.
  • Scalability: Handle 10M+ transactions/month orchestrated via Kubernetes.
  • Observability: Full MLflow tracking and Evidently AI drift detection.

4. Architecture Overview

1. Extraction: Raw Excel ingested into PostgreSQL.
2. Validation: Great Expectations ensures data integrity.
3. Training: Airflow triggers Python scripts → Optuna tunes XGBoost → MLflow logs metrics.
4. Serving: Redis caches predictions → Streamlit queries Redis.
5. Observability: Prometheus scrapes `/metrics` → Grafana visualizes MAPE and Drift.
6. Infrastructure: All containerized in Docker and orchestrated by Kubernetes (`deployment.yaml`).

6. Technical Highlights

  • Model Performance: Optuna optimization boosted the XGBoost Churn AUC-ROC to >0.89, exceeding the 0.88 requirement.
  • MLOps Practices: Evidently AI automatically generates HTML drift reports. Airflow schedules retraining on a 7-day interval.
  • Challenges Faced: Handling non-stationary time-series data with proper decomposition. Balancing model accuracy with interpretability using SHAP.

7. Deployment & Operations

  • Platform: Fully Dockerized local stack, Kubernetes-ready via `k8s/` manifests. Live demo deployed via Streamlit Community Cloud.
  • CI/CD: GitHub Actions pipeline enforces Flake8 linting, Pytest unit testing, and Docker build verifications on push.
  • Monitoring: Prometheus scrapes metrics from the FastAPI `metrics.py` sidecar.
